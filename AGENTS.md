## What this project is
**churro-code** is "an opinionated open-source coding agent client. Run Claude Code, Codex, and more â€” locally or in the cloud" (from README). It's a polyglot Nx monorepo with three apps that share the same product surface:

- `apps/gateway` â€” **ASP.NET Core .NET 10** Web API. Cloud-side service that acts as a NAT gateway between the daemon and clients.
- `apps/desktop` â€” **Electron + Vite + React + TypeScript + Tailwind + shadcn/ui** UI, **managed by Bun**. Local UI on Win/macOS/Linux.
- `apps/daemon` â€” Cross-platform **Go** CLI / installable daemon. Runs on Win/macOS/Linux.

Each app speaks a different stack on purpose; Nx is the unifying task runner.

For per-app guidance (architecture, schema, internal commands), see the app-level `AGENTS.md` when working inside that subtree:
- `apps/desktop/AGENTS.md` â€” Electron app: dockview windowing, tRPC routers, Drizzle schema, release pipeline. (`apps/desktop/CLAUDE.md` is a symlink to it.)
- `apps/gateway/` and `apps/daemon/` do not yet have their own `AGENTS.md`.

## Key architectural conventions
- **One Nx workspace at the repo root.** `nx.json`, `package.json`, `pnpm-workspace.yaml`, `tsconfig.base.json` all live at the root.
- **Two language workspaces, also at root:** `ChurroCode.slnx` (the new XML .NET solution format generated by .NET 10's `dotnet new sln`) and `go.work` (Go workspace; daemon is the only module today, but adding `libs/<x>/go.mod` and `go work use ./libs/<x>` is the path forward).
- **Two package managers, by design:** pnpm at the root (drives Nx + plugins), bun inside `apps/desktop` (the user explicitly wants bun for the Electron app). `pnpm-workspace.yaml` intentionally has `packages: []` so pnpm does NOT manage `apps/desktop/node_modules` â€” bun owns it. Don't re-add `apps/desktop` to the pnpm workspace; the two PMs will fight.
- **Project discovery is plugin-driven, not project.json-driven:**
  - `@nx/dotnet` infers `gateway` from `apps/gateway/Gateway.csproj` (referenced by the root `.slnx`).
  - `@naxodev/gonx` infers `daemon` from `apps/daemon/go.mod` (registered in root `go.work`).
  - `apps/desktop` has a hand-written `project.json` because it shells out to `bun run` for everything (Vite/Electron isn't covered by an Nx plugin we use).
  - The thin `project.json` files in `apps/gateway` and `apps/daemon` exist only to override the inferred project names (lowercase: `gateway`, `daemon`) and add tags. The real targets come from the plugins.
- **No `libs/` yet** â€” create one only when a real shared lib appears. Nx is fine with no libs.

## Common commands

### Whole workspace (run from repo root)
```bash
pnpm install                   # install root deps (Nx + plugins)
pnpm exec nx show projects     # â†’ ["desktop","gateway","daemon"]
pnpm exec nx graph             # interactive project graph
pnpm exec nx run-many -t build # build all three apps; second run is fully cached
pnpm exec nx reset             # clear Nx daemon + cache when project graph seems stale
```

### Working inside `apps/desktop` directly (bun-native)
```bash
cd apps/desktop
bun install                    # NOT pnpm
bun run dev                    # electron-vite dev (with HMR)
bun run build                  # electron-vite build â†’ out/{main,preload,renderer}
bun run typecheck              # tsc on both tsconfig.node.json + tsconfig.web.json
bunx shadcn@latest add <component>   # add more shadcn components
```
Renderer code lives in `src/renderer/` with a `@/*` alias to that directory. Main process is `src/main/index.ts`; preload bridge is `src/preload/index.ts` (sandboxed, contextIsolation on).

### Working inside `apps/gateway` directly (.NET-native)
```bash

dotnet build apps/gateway/Gateway.csproj
dotnet sln ChurroCode.slnx add <new-project>.csproj   # register new .NET projects so @nx/dotnet sees them
```

### Working inside `apps/daemon` directly (Go-native)

## Running tests

### Aggregate (all three apps)
```bash
pnpm exec nx run-many -t test          # run every test target; second run is cached
pnpm exec nx affected -t test          # run only tests affected by local changes
```

### Per-app
| App | Command | Runner | Test files |
|---|---|---|---|
| desktop | `pnpm exec nx run desktop:test` (or `cd apps/desktop && bun run test`) | Vitest 3 | `src/**/*.test.ts` co-located next to source |
| gateway | `pnpm exec nx run gateway:test` (or `dotnet test apps/gateway/Gateway.Tests/Gateway.Tests.csproj`) | xUnit + `WebApplicationFactory` | `apps/gateway/Gateway.Tests/` |
| daemon  | `pnpm exec nx run daemon:test` (or `cd apps/daemon && go test ./...`) | `go test` | `apps/daemon/*_test.go` |

### Desktop test details
Tests run in a Node.js environment (no Electron, no real SQLite). A `vitest.setup.ts` stubs `localStorage` so atoms that use `atomWithStorage({ getOnInit: true })` work in Node. Tests that need a DOM add `// @vitest-environment jsdom` at the top of the file and `import { cleanup } from "@testing-library/react"` + `afterEach(cleanup)` to prevent prior renders from leaking into the next test.

Shared component-test helpers live in [apps/desktop/test-utils/](apps/desktop/test-utils/index.ts): `renderWithProviders` (RTL render with an isolated jotai store), `createTestStore`, `createMockTransport`, `createMockTrpc`. The chat orchestrator's pure decision logic lives under [apps/desktop/src/renderer/features/agents/machines/](apps/desktop/src/renderer/features/agents/machines/) â€” see `apps/desktop/AGENTS.md` for the full layered-architecture + test-battery conventions.

```bash
cd apps/desktop
bun run test              # one-shot run (used by CI and nx)
bun run test:watch        # watch mode for local development
```

### CI
GitHub Actions runs tests on every PR via the existing `.github/workflows/ci.yml` workflow (`nx run-many -t test`). The desktop `bun install` step sets `SKIP_ELECTRON_REBUILD=1` so native-module compilation is skipped in the Linux runner.

## Adding a new project

- **New .NET project:** `dotnet new <template> -n <Name> -o apps/<name>` then `dotnet sln ChurroCode.slnx add apps/<name>/<Name>.csproj`. `@nx/dotnet` will pick it up; add an `apps/<name>/project.json` only if you want to rename or tag it.
- **New Go module:** `cd apps/<name> && go mod init github.com/ChurroStack/churro-code/apps/<name>`, then `go work use ./apps/<name>` from the repo root. `@naxodev/gonx` auto-discovers it.
- **New JS/TS app or lib:** add `<path>/project.json` with explicit `name` and `targets`. If you want pnpm to manage it, add the path to `pnpm-workspace.yaml`.

## Gotchas worth knowing

- **`.slnx` not `.sln`.** .NET 10 generates the new XML solution format. `dotnet sln` works on it the same way; just pass `ChurroCode.slnx` explicitly.
- **`@nx/dotnet` names projects from the `.csproj` filename**, so it called the project `Gateway` until we overrode it to lowercase `gateway` via `apps/gateway/project.json`. Same trick for daemon (`apps/daemon` â†’ `daemon`).
- **`@nx/dotnet` uses `run` as the serve-style target**, even if you set `serveTargetName` in `nx.json`. Use `nx run gateway:run`.
- **Don't `pnpm install` inside `apps/desktop`.** That directory is bun-managed. Always `bun install` there.
- **Peer-dep warnings during root `pnpm install`** (`node-fetch` peer mismatch from `@module-federation/sdk`) are non-fatal â€” Nx still works. Only act if a real `nx` command fails.
- **Drizzle migrations â€” two silent failure modes to watch for:**
  1. **Multi-statement SQL must use `-->statement-breakpoint` separators.** `better-sqlite3`'s `prepare()` rejects any string with more than one statement. When you write a migration with multiple `ALTER TABLE` or `CREATE TABLE` statements, place `-->statement-breakpoint` on its own line between each one. Missing separators cause the migrator to quarantine the database (rename it to `.broken-<timestamp>`) and start fresh â€” all data is lost at next launch.
  2. **The `when` timestamp in `drizzle/meta/_journal.json` must be strictly greater than the previous entry's `when`.** The migrator skips entries whose `when` is â‰¤ the last applied migration's timestamp. A new entry with a timestamp in the past (e.g. any date before April 2026 if the existing entries reach that far) will be silently skipped and its columns will never be added to the DB â€” no error, no warning.
